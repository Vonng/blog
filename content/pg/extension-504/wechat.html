从一个化学扩展说起

两天前，一位用户在 GitHub 上给我提了个 Issue：他在用 RDKit —— 化学信息学领域的事实标准库，能在 PostgreSQL 里做分子结构存储、子结构检索和相似性计算。但他发现 PGDG 官方打包的版本缺了 InChI 功能，折腾了半天，加上编译参数后总算跑通了，但还是希望 Pigsty 能原生支持。

但 RDKit 确实是个硬骨头。大约两年前我就试过一次，想把它收进 Pigsty 的扩展仓库，从 Debian 移植到 EL。结果依赖太多了：Boost、Eigen、RapidJSON、Cairo，外加 InChI、Avalon 等可选模块，每个都有自己的编译开关和版本兼容问题。折腾了一会没跑通，就先搁置了。

但这次不一样。有 Coding Agent 了。

用 Codex / Claude Code 处理这类"构建系统考古"任务简直是降维打击 —— 以前需要反复试错的东西，现在基本一轮对话就能理清。这次发布，把 PGDG 打包到 InChI 支持的问题也一并解决了，本质上就是编译时多开一个标志位再带上 InChI 源码。一把过，用户也很满意。

说实话，看到这种反馈挺开心的。做开源最爽的就是这个时候。

趁热打铁

既然手热了，我就顺便把积压已久的几个"历史疑难杂症"也一起清了。plv8 —— V8 引擎的 PostgreSQL 绑定，之前在 EL10 上死活编译不过，这次打了好几个补丁终于搞定了稳定构建。duckdb_fdw —— 允许从 PG 内部读写外部 DuckDB 文件，但之前会和 DuckDB 官方的 pg_duckdb 扩展争抢共享库，我只能忍痛临时隐藏。这次把 duckdb_fdw 挂成了 pg_duckdb 的子扩展，共享同一份 libduckdb，冲突问题优雅解决，俩扩展又能并存了。

然后我就想：既然工具链都热好了，不如把 PostgreSQL 生态里剩下那些值得收录但一直没啃的扩展也一并搞进来吧。于是就有了这次的大更新 —— 新增 32 个扩展，更新 22 个，Pigsty 扩展仓库总数正式突破 500，达到 504 个。

基本上，Pigsty 在这个赛道上已经做到了前无古人，后无来者了。

新扩展

这批新增扩展的画风相当硬核。按大类可以分四组：

数据域 把化学分子、RDF 三元组、BSON、Protobuf、循环日程这些"复杂对象"变成数据库一等公民
查询力 稀疏线代与图算法、Datalog 图查询、全文检索、混合排序融合、递归 SQL 模板引擎
生产线 深度可观测性、查询遥测导出、CDC 到 MQTT、COPY 命令拦截、DDL 逻辑复制补全、轻量分布式锁
开发者 会话变量、伪自治事务日志、自然语言时间解析，以及压轴的 —— 在数据库里跑 TypeScript

这些扩展共同指向一个趋势：PostgreSQL 的扩展层正在把数据库推向应用与数据平台的中间地带。很多原本需要独立服务才能解决的问题，现在可以在一条 SQL 事务边界内搞定。

以下逐一介绍这 33 个新成员。（以下为 Claude/Codex/Gemini 三剑客研究报告汇总）

1 rdkit —— 把化学信息学搬进 PostgreSQL

RDKit 是开源化学信息学领域的事实标准库，由 Greg Landrum 发起（最初在 Novartis，现属 T5 Informatics），其 PostgreSQL cartridge 模块将分子结构存储、子结构检索和相似性计算直接带入关系数据库。对于制药公司和化学研究机构而言，这意味着可以用标准 SQL 查询数百万化合物，无需借助外部工具链。

RDKit cartridge 引入了两组核心数据类型：mol（分子）和 qmol（查询分子，即 SMARTS 模式），以及 bfp/sfp（位指纹/稀疏指纹）。操作符方面，@> 用于子结构匹配，% 用于 Tanimoto 相似性判断，<%> 作为距离运算符。所有这些操作都可以通过 GiST 索引加速——索引内部基于指纹筛选进行快速预过滤，再做精确匹配。

以 ChEMBL 数据库（187 万化合物）为例：

SQL 
-- 子结构检索：查找含有特定骨架的分子
SELECT count(*) FROM rdk.mols WHERE m @> 'c1cccc2c1nncc2';
-- 结果：461 个匹配，耗时约 108ms

-- Tanimoto 相似性搜索：基于 Morgan 指纹
SELECT molregno,
  tanimoto_sml(morganbv_fp(
    mol_from_smiles('c1ccccc1C(=O)NC'::cstring)), mfp2
  ) AS similarity
FROM rdk.fps JOIN rdk.mols USING (molregno)
WHERE morganbv_fp(mol_from_smiles(
  'c1ccccc1C(=O)NC'::cstring)) % mfp2
ORDER BY morganbv_fp(mol_from_smiles(
  'c1ccccc1C(=O)NC'::cstring)) <%> mfp2;

-- SMARTS 模式匹配
SELECT * FROM rdk.mols
WHERE m @> 'c1[o,s]ncn1'::qmol LIMIT 500;

应用场景集中在药物研发的几个关键环节：先导化合物骨架搜索（在百万级化合物库中做子结构匹配）、SAR 分析（通过相似性检索寻找活性类似物）、化合物注册系统（利用结构指纹做重复性检查）、以及商业化合物目录检索（如 eMolecules 的 600 万+化合物数据集）。

工程落地时需要注意：cartridge 的重点不在"能不能算"，而在"能不能被索引、能不能被 planner 正确利用"。在 187 万化合物上子结构检索耗时在 88ms 至 1900ms 之间，经过优化可处理 600 万+化合物规模的数据集。BSD 许可证。
2 provsql —— 半环溯源让查询结果"可追溯"

ProvSQL 由巴黎高等师范学校教授 Pierre Senellart 和 INRIA Valda 团队开发，发表于 VLDB 2018。它为 PostgreSQL 添加 (m-)半环溯源（semiring provenance）和不确定性管理——能自动追踪每个查询结果是由哪些基础元组"推导"出来的，并支持在不同代数结构（布尔、安全等级、计数、概率）下对溯源信息进行求值。

SQL 
-- 安全等级传播：结果自动继承最高安全级别
SELECT create_provenance_mapping(
  'personnel_level', 'personnel', 'classification');

SELECT p1.city,
  security(provenance(), 'personnel_level')
FROM personnel p1, personnel p2
WHERE p1.city = p2.city AND p1.id < p2.id
GROUP BY p1.city ORDER BY p1.city;

-- 布尔公式溯源
SELECT *, formula(provenance(), 'witness_mapping') FROM s;

-- 概率查询
SELECT city, probability_evaluate(provenance())
FROM result;
← 左右滑动查看 →

适合四类场景：安全分级传播——查询结果自动继承源数据中最高的安全等级；概率数据库——计算查询结果的正确概率；数据血缘审计——精确追踪每个结果行的来源；可信度评估——例如在刑事调查中通过溯源加权评估可靠性。

ProvSQL 的价值体现在"可组合性"：溯源结果不是字符串日志，而是可以继续被函数处理的对象。建议用于关键链路，而非全库无差别开启。C/C++ 实现，支持 PG 10–18，MIT 许可证。
3 onesparse —— 在 SQL 里跑十亿边级图算法

OneSparse 将高性能稀疏线性代数带入 PostgreSQL，封装了 SuiteSparse:GraphBLAS 库。核心理念是将图表示为稀疏矩阵，用矩阵乘法实现 BFS、PageRank、三角中心性等图算法——而这一切都在 SQL 中完成。

SQL 
-- 从 Matrix Market 文件加载图
SELECT mmread('/home/postgres/onesparse/demo/karate.mtx') AS graph;

-- BFS 遍历
SELECT (bfs(graph, 1)).level FROM karate;

-- 度中心性（矩阵列归约）
SELECT reduce_cols(cast_to(graph, 'int32')) AS degree FROM karate;

-- PageRank
SELECT pagerank(graph) FROM karate;

在 GAP benchmark 上，对 43 亿边 的图执行 BFS 时达到 70 亿+边/秒 的吞吐量（48 核 AMD EPYC 服务器）。应用场景包括金融反欺诈（交易网络环检测）、社交网络分析、Graph RAG 等。

OneSparse 要求 PG 18 Beta 或更新版本，当前处于 Alpha 阶段。Apache 2.0 许可证。

5 datasketches —— Apache 出品的亿级近似分析利器

Apache DataSketches 将多种近似分析数据结构（Sketch）引入 SQL 世界。核心问题很明确：在海量数据上做精确的 COUNT(DISTINCT)、分位数计算和频繁项统计太慢或太耗内存。

提供七种 Sketch 类型：cpc_sketch（去重计数）、hll_sketch（HyperLogLog）、theta_sketch（支持集合交并差）、kll_float_sketch（分位数估算）等。关键点不是"有个函数返回估计值"，而是 Sketch 作为可序列化对象可以被聚合合并。

SQL 
-- 近似去重计数：比 COUNT(DISTINCT) 快约 6 倍
SELECT cpc_sketch_distinct(id) FROM random_ints_100m;
-- 结果：63423695（精确值：63208457），~20s vs ~2min

-- Theta Sketch 集合运算：两个用户群体的交集
SELECT theta_sketch_get_estimate(
  theta_sketch_intersection(sketch1, sketch2)
) FROM theta_set_op_test;

-- KLL 分位数估算：获取中位数
SELECT kll_float_sketch_get_quantile(sketch, 0.5)
FROM kll_float_sketch_test;
← 左右滑动查看 →

在 1 亿行数据上，CPC Sketch 去重计数约 20 秒完成（精确 COUNT(DISTINCT) 约 2 分钟）——速度提升 6 倍，相对误差在个位数百分比范围内。

7 pg_stat_ch —— ClickHouse 官方出品的查询遥测

pg_stat_ch 由 ClickHouse 公司开发并开源。与 pg_stat_statements 在 PostgreSQL 内部做聚合统计不同，pg_stat_ch 将每条查询的原始执行事件（45 个字段、固定 4.6KB）实时流式导出到 ClickHouse，让所有聚合分析在 ClickHouse 的分析引擎中完成。

数据管道架构：PostgreSQL Hooks → 共享内存环形缓冲区 → 后台 Worker → ClickHouse。使用 ClickHouse 原生二进制协议加 LZ4 压缩。

SQL 
-- PostgreSQL 侧：监控扩展健康状态
SELECT * FROM pg_stat_ch_stats();
-- 返回：enqueued, exported, dropped 计数

-- ClickHouse 侧：过去 1 小时按应用统计 p95/p99
SELECT query_id, count() AS calls,
  quantile(0.95)(duration_us) / 1000 AS p95_ms,
  quantile(0.99)(duration_us) / 1000 AS p99_ms
FROM pg_stat_ch.events_raw
WHERE app = 'myapp'
  AND ts_start > now() - INTERVAL 1 HOUR
GROUP BY query_id
ORDER BY p99_ms DESC LIMIT 10;

性能令人印象深刻：p99 开销 ~5μs/条，在 pgbench 32 客户端 36.6K TPS 下，30 秒内捕获 770 万事件且零丢弃，对 TPS 影响 <1%。支持 PG 16–18，Apache 2.0 许可证。

8 pg_rrf —— 一个函数搞定混合检索的排序融合

pg_rrf 将 Reciprocal Rank Fusion（RRF）封装为原生 PostgreSQL 函数，解决混合检索场景中"分数不可比"的工程痛点。公式为 score(d) = Σ 1/(k + rank_i(d))，k 默认 60。

SQL 
-- 使用 pg_rrf 的混合检索：pgvector + BM25
WITH fused AS (
  SELECT * FROM rrf_fuse(
    ARRAY(SELECT id FROM docs
      ORDER BY bm25_score DESC LIMIT 100),
    ARRAY(SELECT id FROM docs
      ORDER BY embedding <=> :qvec LIMIT 100),
    60
  )
)
SELECT d.*, fused.score
FROM fused JOIN docs d USING (id)
ORDER BY fused.score DESC LIMIT 20;

这将原本需要 FULL OUTER JOIN + COALESCE 链 + 手动评分公式的 20+ 行 CTE 压缩为一个函数调用。把融合逻辑移到数据库侧，减少了应用层拼接的开销。MIT 许可证。

9 pg_kazsearch —— 哈萨克语全文检索的"从无到有"

SQL

-- {алма}

-- 带权重检索
SELECT title FROM articles
WHERE fts @@ websearch_to_tsquery(
  'kazakh_cfg', 'президенттің жарлығы')
ORDER BY ts_rank_cd(fts, websearch_to_tsquery(
  'kazakh_cfg', 'президенттің жарлығы')) DESC
LIMIT 10;

10 pg_liquid —— Datalog 风格的图查询

pg_liquid 把 Liquid/Datalog 风格的声明式图查询带进 PostgreSQL。支持事实断言、递归传递闭包、复合查询和行规范化器。

SQL 
SELECT target
FROM liquid.query($$
  Edge("a", "path", "b").
  Edge("b", "path", "c").
  Edge("c", "path", "d").

  Reach(x, y) :- Edge(x, "path", y).
  Reach(x, z) :- Reach(x, y), Reach(y, z).

  Reach("a", target)?
$$) AS t(target text)
ORDER BY 1;

11 logical_ddl —— 让逻辑复制也能同步 DDL

PostgreSQL 的逻辑复制只处理 DML，不处理 DDL，表结构不同步会直接中断复制。logical_ddl 通过事件触发器拦截 DDL 命令，将其解析并保存到复制表中，订阅端接收后生成等效 SQL 并执行。

SQL 
-- 发布端配置
INSERT INTO logical_ddl.settings (publish, source)
VALUES (true, 'publisher1');

-- 将逻辑复制中的所有表加入 DDL 追踪
INSERT INTO logical_ddl.publish_tablelist (relid)
SELECT prrelid FROM pg_catalog.pg_publication_rel;

-- 按表指定捕获的 DDL 类型
INSERT INTO logical_ddl.publish_tablelist (relid, cmd_list)
VALUES ('my_table'::regclass,
  ARRAY['ADD COLUMN', 'DROP COLUMN']);

13 pgbson —— 比 JSONB 更精确的二进制文档类型

为 PostgreSQL 引入原生 BSON 数据类型。BSON 提供一等公民的 datetime、decimal128、int32/int64、binary 等类型，保证二进制完美往返。配合函数索引可实现 10,000 倍 的查询加速。

SQL 
-- 插入带丰富类型的 EJSON 文档
INSERT INTO data_collection (data) VALUES (
  '{"d":{"recordId":"R1",
    "amt":{"$numberDecimal":"77777809838.97"},
    "ts":{"$date":"2022-03-03T12:13:14.789Z"}}}');

-- 函数索引 + dotpath 查询（推荐方式）
CREATE INDEX ON data_collection(
  bson_get_string(data, 'd.recordId'));

SELECT bson_get_decimal128(data, 'd.amt')
FROM data_collection
WHERE bson_get_string(data, 'd.recordId') = 'R1';

15 pgmqtt —— 数据库变更直推 MQTT

把 PostgreSQL 的变更（INSERT/UPDATE/DELETE）通过 CDC 直接变成 MQTT 消息推给订阅者，同时支持入站消息按映射写回表。IoT 场景尤其合适——无需外部中间件即可将数据库状态变化推送到边缘设备。

SQL

33 pg_typescript —— 在 PostgreSQL 里写 TypeScript

本次盘点的压轴亮点：为 PostgreSQL 添加 TypeScript 作为一等过程语言（LANGUAGE typescript），底层由 Deno 的 API 和 V8 运行时驱动。支持完整的 TypeScript 类型标注、原生 async/await、npm 包导入、以及 Deno 的细粒度权限模型。

架构设计

每个 PostgreSQL 后端进程维护一个独立的 Deno 运行时和 Tokio 异步运行时。导入的 npm 模块在 CREATE FUNCTION 时预取并存储在 PostgreSQL 表中，而非运行时动态加载。编译后的函数句柄在后端内做缓存。

权限系统

继承 Deno 的"默认拒绝"安全模型：网络、文件系统、环境变量、子进程等能力全部默认关闭，通过 GUC 参数逐项授权。超级用户可设置 typescript.max_allow_* 作为上限。

SQL + TypeScript 
-- 导入 npm 包生成 URL slug
CREATE FUNCTION slugify(title text) RETURNS text
LANGUAGE typescript
SET "typescript.import_map" =
  '{"imports":{"lodash":"https://esm.sh/lodash@4"}}'
AS $$
  return lodash.kebabCase(title);
$$;
SELECT slugify('Hello World');  -- 'hello-world'

-- 在 TypeScript 中执行 SQL
CREATE FUNCTION active_user_emails() RETURNS jsonb
LANGUAGE typescript
SET "typescript.allow_pg_execute" = 'on'
AS $$
  const result = await _pg.execute(
    "SELECT email FROM users WHERE active = $1",
    [true]
  );
  return result.rows.map(r => r.email);
$$;

-- 网络访问：调用外部 API（域名白名单）
CREATE FUNCTION call_api(url text) RETURNS jsonb
LANGUAGE typescript
SET "typescript.allow_net" = 'api.example.com'
AS $$
  const res = await fetch(url);
  return await res.json();
$$;

与 plv8 的对比

pg_typescript 在几个关键维度上超越 plv8：语言——直接支持 TypeScript 而非仅 JavaScript；异步——原生 async/await；模块——通过 import map 导入 npm 包；安全——完整的 Deno 权限体系；API——fetch() + _pg.execute()。

pg_typescript 当前处于 Alpha 阶段。V8 冷启动存在延迟，建议加入 shared_preload_libraries 预热。支持 PG 16–18，Rust（pgrx）+ rusty_v8 构建，MIT 许可证。

趋势总结
